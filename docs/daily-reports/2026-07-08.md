## What changed

- **Second review-fix round** (this commit): the run-claim in
  `run_drive_sync` is now atomic (`filter(status=QUEUED).update(RUNNING)`) so
  duplicate Celery deliveries can never double-execute a sync; Drive's md5
  moved to its own `SourceDocument.drive_md5_checksum` field (migration 0003)
  so `content_hash` is owned exclusively by the content stage — closing the
  remaining overwrite paths (metadata-only syncs, exclusion transitions);
  throttle tests clear the cache on teardown; documented that
  `DJANGO_CACHE_URL` must move together with `REDIS_URL`. Tests 36 → 40.
- **ADR-008 and ADR-009** (`ai-context/04-decisions.md`): single-tenant
  compose stack per client; per-client service account provisioned by us
  ("share to connect" onboarding, no client-side GCP work, no per-user
  OAuth); folder/shared-drive picker is now part of the current Phase 2
  onboarding path, including the root-change re-scope rule. This resolves
  the previously open delegation-vs-OAuth question and removes manual root IDs
  as the client-facing setup path.
- **Compose: service-account key mount** (`infra/compose.app.yml`): the key
  is bind-mounted read-only into `django` and `celery-worker` at
  `/run/secrets/google-service-account.json`, with the container-side env
  override; `/dev/null` default keeps the stack bootable before the key
  exists. Combined compose config validated.
- **Admin Drive root selection flow:** added Drive root discovery for folders
  shared with the service account plus shared drives, and admin-only endpoints
  to list visible roots and persist the chosen folder/shared-drive scope in
  `DriveConnection`. Sync still reads only from server-side connection state.
- **Review tooling adjustment:** removed the `agy` fallback from the local
  review scripts. Stage 2 now uses Claude Code CLI when available; otherwise
  `REVIEW.md` records a static-check-only result.
- **Drive root-selection hardening:** root discovery failures now return
  controlled API responses, root selection saves under a database row lock, and
  root changes fail closed by marking existing source documents non-retrievable
  until the new scope is synced.
- **Review-gate follow-up on root discovery:** the previous commit's gate
  flagged that moving `build_drive_service` out of the `try` in
  `list_root_candidates` let service-construction failures (revoked keys,
  discovery errors) escape as raw 500s. The build call is back inside the
  error boundary with an explicit `MissingServiceAccountKeyError` passthrough
  (409 vs 502 unchanged), pinned by a test that patches construction to raise
  `RefreshError` and asserts the controlled 502 body. Also from that review:
  anonymous POSTs to the root-selection endpoint now have their own test, the
  duplicated non-admin test setup moved into a `_login_member` helper, and the
  `MissingServiceAccountKeyError` docstring now says it covers malformed and
  unreadable key files, not just missing ones. Tests 61 → 63.
- **Drive permission-fetch failure handling:** live validation showed files can
  be listable through a shared folder while `permissions.list()` fails per file.
  The Drive client now isolates that failure to the affected file, marks the
  metadata as incomplete, and the sync path excludes that document from
  retrieval instead of crashing the run or treating an empty permission list as
  safe. Tests 63 → 65.
- **Retrieval-path verification:** checked backend code paths for retrieval,
  Neo4j query, chunk lookup, and LLM context assembly. None are implemented yet;
  Phase 5 now explicitly requires `retrieval_eligible = false` to exclude all
  chunks, graph facts, citations, and prompt context even if stale SpiceDB
  relationships exist.

## Follow-up fix round (external review + gate notes)

An external commit-by-commit review of the pushed repo (origin/main, through
`212f521`) plus the gate's APPROVE-WITH-NOTES items from `2ce36d5` produced
one batch of fixes, all verified against the code first:

- **Makefile `--env-file` fix (gate finding, real bug):** compose resolves
  `${...}` interpolation from `infra/.env`, not the repo root, so the
  service-account key mount silently degraded to `/dev/null` under the real
  `make up`. `COMPOSE` now passes `--env-file .env` when the file exists.
  Verified: `docker compose config` shows the real host key path.
- **Fail-loud missing key:** `build_drive_service` raises
  `MissingServiceAccountKeyError` when the key path is unset/missing/empty
  (the `/dev/null` bootstrap mount), so `error_summary` names the actual
  problem instead of an opaque credential parse error.
- **Double RUNNING write (gate finding):** `sync_drive_metadata` no longer
  re-transitions a run the Celery task already claimed, so the claim's
  `started_at` survives.
- **Ops ports bypassed ops-auth (external review, most important):** Dozzle
  (`9999`), Uptime Kuma (`3001`), and the unauthenticated Traefik dashboard
  (`8088`) were published on all interfaces, skipping the basicauth
  middleware entirely; Open WebUI (`3000`) included for consistency. All are
  now `127.0.0.1`-bound with env-var overrides — remote access goes through
  Traefik.
- **Default ops credential fails loudly in production:** settings.py refuses
  to boot outside DEBUG/tests when `TRAEFIK_OPS_BASIC_AUTH_USERS` is unset or
  still the documented admin/password default — same treatment as
  `DJANGO_SECRET_KEY`.
- **Smoke endpoint gated:** `SmokeTaskView` was `AllowAny` with no auth —
  anyone could queue Celery work. Now `IsAdminUser`, same bar as the sync
  endpoint; `make smoke` enqueues directly via `manage.py shell` instead.
- The gate's third note (stale `content_hash` dataclass references after the
  rename) was checked and is a non-issue — the only remaining write is the
  model field in the content stage, which is correct.

Gate follow-through: the secret scan now exempts the exact fixture literal
`test-password` (it blocked this commit as a false positive); a real ops
basicauth credential was generated into the local `.env` (user `ops`) since
the stack runs with `DEBUG=False` and would otherwise refuse to boot.

Backlog note (per gate suggestion): the atomic claim moved the RUNNING
transition to the moment of claim, *before* any work starts, so the
stale-RUNNING window (worker killed mid-run leaves the row stuck) is now
slightly wider than before. The Phase 3 sweep item should account for that.
