## What changed

- **A — delegated-subject env fallback removed** (`drive/google_client.py`):
  `build_drive_service` resolved the delegation subject as
  `connection.delegated_subject_email or settings.GOOGLE_DRIVE_DELEGATED_SUBJECT`.
  Clearing the subject via the admin endpoint reported `""` and invalidated
  documents while the built service *still* delegated to the env subject — an
  auth-identity mismatch. The connection field is now authoritative (it is
  already seeded from the env var at bootstrap), so clearing genuinely clears.
- **C — GET Drive endpoints no longer mutate** (`integrations/views.py`):
  `_active_or_bootstrap_connection()` created a `DriveConnection` row on read.
  Root discovery (`GET .../drive/roots/`) now answers from a transient
  (unsaved) connection built from settings; the permission-check diagnostic
  (`GET .../drive/permissions/check/`) returns 409 when no connection exists.
  Create-on-write is retained only for the POST selection/subject endpoints.
- **B — distinct `listing_failed` status** (`integrations/views.py`): the
  permission-check summary previously reported `blocked` both when ACL samples
  were unreadable and when folder listing itself failed. A `listing_failed`
  label now separates "couldn't walk the tree" from "ACLs unreadable" so the
  summary alone points at the right remediation (`folder_listing_errors`
  already carried the count).
- **D — shared BFS folder walk** (`drive/google_client.py`): `list_files` and
  the permission diagnostic duplicated the breadth-first queue + dedup
  scaffolding. Extracted `_walk_files(...)` with an `on_listing_error` hook so
  the two intentional error semantics stay explicit — content sync propagates a
  folder-listing failure (fails closed), the diagnostic counts it and
  continues. Behavior-preserving; no extra Drive API call on the diagnostic
  path (root path is passed in, not re-resolved).

Tests 85 → 88 (backend suite): added `build_drive_service` subject tests (A),
non-mutating GET tests for both read endpoints (C), and re-pointed the
folder-listing status test to `listing_failed` (B). `make lint` clean.

## Extraction engine adopted: neo4j-graphrag (this session, latest)

Researched the field (neo4j-graphrag, Graphiti, Microsoft GraphRAG,
LightRAG/cognee, LlamaIndex PropertyGraphIndex) instead of building
extraction from scratch; chose the official `neo4j-graphrag` package —
**ADR-010**, which also resolves the open fact-level vs document-level
provenance question (fact-level is supported and adopted). Implemented on
`phase-3/graph-foundation` (still unpushed):

- `graph/graphrag.py`: adapter wrapping only the package's extraction
  component, driven chunk-by-chunk for exact `chunk_index` provenance; the
  ontology is passed as a closed engine schema, but enforcement stays at
  our `validate_extraction_result` boundary. LLM via OpenRouter
  (`OpenAILLM` with base_url override).
- Entity/relationship writers (`replace_document_entities`): entities are
  **per-document scoped** (ADR-010 — cross-document merging is a permission
  hazard), stored under one `:Entity` label (`entity_type` as property, to
  avoid colliding with the structural `:Document` constraint), each with a
  `mentions` edge to its source chunk; relationship endpoints resolved by
  name within the document, unresolvable/ambiguous ones counted and
  skipped, missing chunk anchors raise.
- `GRAPH_EXTRACTION_ENGINE` setting (`paragraph` default | `neo4j_graphrag`,
  validated at startup; graphrag mode refuses to boot without
  `OPENROUTER_API_KEY` + `GRAPH_EXTRACTION_MODEL`). `.env.example` updated.
- Suite 120 → 133; the graphrag tests exercise the real extraction
  component (prompting, JSON parsing, validation) with only the LLM faked.
