## Independent branch review and hardening (Claude session)

- **What changed:** a second full review of the Phase 3 branch confirmed the
  earlier fixes and found the remaining recovery/quality gaps, which are now
  fixed:
  - Stale-RUNNING sweeper (`integrations.sweep_stale_graph_extractions`, beat
    every 15 min): a worker lost after the broker dropped its message no
    longer strands a document in RUNNING forever.
  - Bounded cross-sync retries: `graph_extraction_attempts` (migration 0005)
    counts FAILED transitions per content version;
    `GRAPH_EXTRACTION_MAX_SYNC_ATTEMPTS` (default 5) caps sync-driven
    requeues so a deterministic failure cannot burn LLM calls on every sync.
  - PENDING requeue gate: `graph_extraction_queued_at` distinguishes "just
    queued" from "message evidently lost"
    (`GRAPH_EXTRACTION_PENDING_REQUEUE_AFTER_MINUTES`, default 15), so
    overlapping syncs no longer double-enqueue the same extraction.
  - Relationship endpoints now resolve by (entity_type, name): the GraphRAG
    adapter carries the LLM's node labels through, so an edge between
    same-named entities of different types (Person "Mercury" vs Topic
    "Mercury") is written instead of dropped as ambiguous.
  - Retryable-exception classification moved behind the engine boundary
    (`graph.pipeline.get_retryable_extraction_exceptions`); the Celery task
    no longer imports openai, and a future engine brings its own transient
    error set.
  - Writer batching: chunks, entity mentions, and relationships now write via
    UNWIND (3 + one-per-relationship-type round-trips instead of one per
    row); per-chunk LLM extraction runs concurrently
    (semaphore-bounded, `GRAPH_EXTRACTION_MAX_CONCURRENT_LLM_CALLS`,
    default 4).
  - `graph.guard.provenance_where` now validates its alias at the
    interpolation point.
  - Fixed the stale test assertion left by the content-hash work
    (`RunDriveSyncTaskTests`), which had the working tree at 1 failing test.
