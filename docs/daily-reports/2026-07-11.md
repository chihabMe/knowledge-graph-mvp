## Phase 4 review-fix session

### What changed

- Ran a high-effort multi-angle review of the Phase 4 branch (8 finder
  angles + verification) and fixed all 10 surviving findings in 10 focused
  commits, 195 tests green:
  - Closed the group-revocation gap: beat-scheduled permission syncs
    (`PERMISSION_SYNC_INTERVAL_SECONDS`) now bound how long a removed group
    member can retain access, since membership changes never alter a
    document's ACL hash.
  - Removed the run-start eligibility wipe (every sync was a full retrieval
    blackout; a lost worker made it permanent) and added a
    `sweep_stale_permission_sync_runs` beat sweeper.
  - Excluded principals with null `emailAddress` instead of crashing the run,
    and excluded documents with zero effective grant paths
    (`no_effective_grants`, migration 0007) so eligibility always implies a
    verified SpiceDB grant.
  - Keyed the Drive-root readiness check to `scope_type` via
    `DriveConnection.effective_root_id`, shared by views and client.
  - Added `SPICEDB_GRPC_TLS` with a production guard
    (`SPICEDB_GRPC_ALLOW_INSECURE` opt-out) — the channel was always
    plaintext.
  - Scoped SpiceDB tuple reads server-side to the connection prefix and
    pinned `MANAGED_RESOURCE_TYPES` to schema.zed with a test.
  - Single-sourced the permission-verified predicate
    (`SourceDocument.objects.permission_verified()` + instance twin).
  - Reused `_walk_files` for the permission scan (no per-folder `files().get`).
  - Bounded the SpiceDB health probe to 1s on one shared channel.

### Decisions

- A failed permission run now keeps the previous verified state instead of
  blanking the connection; the fully consistent SpiceDB lookup remains the
  authoritative query-time gate, so revocations still apply immediately once
  reconciled.
- Group-revocation staleness is bounded by the permission-sync beat interval
  (default 900s), documented in `docs/permission-sync.md`.
