# Daily Report — 2026-07-12

## What changed

Phase 4 hardening day (planned Day 2 of 3): applied the seven deferred
cleanups from the review, one commit each, 198 tests green after each
commit.

- Merged the two permission-snapshot models onto one abstract base and
  dropped their never-read `source_permissions_version` copies
  (migration 0008); `_snapshot()` now catches `ObjectDoesNotExist` once.
- Extracted one reason-coded service-account credential loader
  (`integrations/drive/credentials.py`) behind both the Drive and Directory
  clients; domain error types and messages are byte-identical.
- Deduplicated the capped exponential retry countdown into
  `_retry_countdown()` (three call sites).
- Removed the `"fully_consistent"` sentinel from `spicedb_revision`; empty
  now uniformly means "no explicit revision" (safe because eligible
  documents always have grant tuples, hence a real write token).
- `allowed_source_document_ids` logs the exception class when it fails
  closed — a SpiceDB outage is no longer indistinguishable from a user
  with no access. Never logs principals or payloads.
- The per-connection sync lock TTL now derives from
  `PERMISSION_SYNC_STALE_RUN_TIMEOUT_MINUTES` instead of a fixed 900s that
  matched the beat interval exactly — a >15-minute scan would have lost the
  lock mid-flight and collided with the next scheduled run.
- The folder cycle validator and nested-group resolver walk iteratively;
  1500-deep chain tests prove client data can no longer trigger
  `RecursionError`.

## Decisions

- Credential-loading failures stay reason-coded internally and mapped to
  per-client domain errors at the boundary, preserving the exact existing
  messages (Drive: three specific texts; Directory: one controlled text).
- Traversal depth is treated as client-controlled data, so recursion is
  banned from both hierarchy walks.

## Day 3: live-stack validation and closeout

- Brought up the core + scheduler compose services (`postgres`, `redis`,
  `neo4j`, `spicedb`, `django`, `celery-worker`, `celery-beat`,
  `spicedb-schema`); skipped `traefik`/`open-webui`/`dozzle`/`uptime-kuma`,
  which aren't required for this checklist and hit repeated registry
  timeouts (Docker Hub, GHCR) unrelated to the app.
- `spicedb-schema` (and, by extension, every non-TLS SpiceDB caller) failed
  closed on first boot: `grpcutil.insecure_bearer_token_credentials()`
  composes the bearer token with
  `grpc.local_channel_credentials(LocalConnectionType.LOCAL_TCP)`, which
  gRPC's transport layer refuses for any address but loopback. Every
  container reaches SpiceDB at its compose service address, never
  `127.0.0.1`, so `SPICEDB_GRPC_ALLOW_INSECURE=true` could never actually
  produce a working connection in this topology despite acknowledging
  exactly that private-network deployment. Fixed by routing the bearer
  token through a small gRPC interceptor over a plain
  `grpc.insecure_channel` instead (`apps/backend/authorization/client.py`);
  added a regression test pinning the old helper as unused on that path.
  199 tests green, lint clean.
- Also had to generate a real `DJANGO_SECRET_KEY` and
  `SPICEDB_GRPC_PRESHARED_KEY` and set `SPICEDB_GRPC_ALLOW_INSECURE=true` in
  the local `.env` — `spicedb-schema` always builds against the prod Docker
  target (no `DJANGO_DEBUG` override), so local dev has to satisfy the same
  hardened settings checks as production.
- Applied migrations 0006–0008 (0006 was also still pending, ahead of the
  0007/0008 the handoff named).
- Live-validated the full checklist: `spicedb-schema` applies cleanly;
  `/api/health/` returns 200 with all services `ok`, and 503 "degraded"
  (spicedb: error, no leaked detail) within ~1.1s of stopping SpiceDB;
  `allowed_source_document_ids` returns `()` and logs the
  `authorization.lookup` warning with only the exception class name while
  SpiceDB is down; `manage.py check` refuses to boot with `DJANGO_DEBUG=false`
  and neither `SPICEDB_GRPC_TLS` nor `SPICEDB_GRPC_ALLOW_INSECURE` set, and
  boots clean once the ack is present.
- Watched the beat-scheduled permission sync fire live (no manual POST): a
  fresh persistent-schedule file treated the periodic tasks as immediately
  due, so `schedule_permission_syncs` and `sweep_stale_permission_sync_runs`

  full 900s interval. `run_permission_sync` created `PermissionSyncRun`

- Merged `phase-4/spicedb-permissions` into `main` and deleted the merged
