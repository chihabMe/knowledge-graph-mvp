## What changed

cleanups from the review, one commit each, 198 tests green after each

- Merged the two permission-snapshot models onto one abstract base and
  dropped their never-read `source_permissions_version` copies
  (migration 0008); `_snapshot()` now catches `ObjectDoesNotExist` once.

  `PERMISSION_SYNC_STALE_RUN_TIMEOUT_MINUTES` instead of a fixed 900s that
  matched the beat interval exactly — a >15-minute scan would have lost the
  lock mid-flight and collided with the next scheduled run.

  1500-deep chain tests prove client data can no longer trigger
  `RecursionError`.

## Post-merge review pass

A second-model review of the merged Phase 4 work confirmed the fail-closed
contract intact across all eight code commits and found two defects, both
fixed the same day (201 tests green, lint clean, schema check re-validated
live against the running SpiceDB container):

- The lock-TTL fix had exposed a scheduling deadlock: a beat run colliding
  with a long scan exhausts its lock retries in ~14s, leaving its row QUEUED
  forever with no task message — and `schedule_permission_syncs` skipped any
  connection with a QUEUED run while the sweeper only fails RUNNING ones, so
  one >15-minute scan permanently killed that connection's scheduled syncs
  (unbounded revocation staleness). The scheduler now re-dispatches orphaned
  QUEUED runs instead of skipping; re-sending is idempotent through the
  status CAS and the per-connection lock.
- The insecure-channel fix had hand-rolled ~50 lines (bearer interceptor +
  a `Client.__new__` `__init__` bypass) that authzed 1.24.4 already ships as
  `authzed.api.v1.InsecureClient` — its docstring names the docker-compose
  non-loopback case verbatim. Swapped to the library client; the loopback
  regression test still pins `insecure_bearer_token_credentials` as unused.

A follow-up full-surface audit of the Phase 4 authorization subsystem
(schema, reconciler, identifiers, lookup gate, Drive ACL parsing, group
resolution, tasks, guards) found no permission leaks — every traced failure
path lands closed — but two real fail-closed bugs, both verified by writing
the regression test first and watching it fail:

- `_referenced_groups` forwarded `deleted: true` ACL entries to the
  Directory resolver (its siblings skip them), so one deleted Workspace
  group lingering in one file's ACL 404'd resolution and excluded every
  group-granted document on the connection until the ACL was hand-cleaned.
- `_document_reason` kept INACTIVE_IN_SCOPE sticky even though every
  document it examines was just re-activated by the running scan, so a
  document moved out of the synced root and back was re-excluded on every
  permission run indefinitely (content syncs, which clear the reason, are
  admin-triggered, not scheduled).

Known residual (accepted, not gating): group resolution stays
all-or-nothing, so a group that 404s *without* a `deleted` ACL flag would
still degrade the whole connection; per-group failure granularity is a
possible Phase 5+ hardening. `spicedb_revision` remains write-only evidence
for the Phase 5 consistency story.
