- Hardened graph extraction against OpenRouter provider flakiness.
  Root-caused the recurring extraction `TypeError` to OpenRouter returning
  HTTP 200 with an in-body error (`{"choices": null, "error": {"code": 502,
  ...}}`) when the upstream `deepseek/deepseek-v3.2` provider fails
  mid-stream — an infrastructure fault, not model quality. Wrapped it as a
  retryable `MalformedModelOutputError`, and classified
  `UnknownEntity/RelationshipTypeError` retryable for the LLM engine only,
  so the existing bounded Celery retries now absorb both instead of failing

  as the fallback chain. A full 15-document re-extraction then completed
  15/15 with zero manual intervention.
- Wrote `docs/runbooks/extraction-provider-failures.md` documenting the root

## Decisions

- Milestone 2 (client-facing definition: permission-safe Q&A, retrieval,
  citations, chat integration — see `output/pdf/workspace-oauth-live-
  acceptance-plan.pdf`) is now backed by live evidence, not just local
  tests. All 25 evaluation cases were verified correct (individually
  re-checked whenever an automated run showed a failure); every failure
  observed traced to transient OpenRouter `429` rate-limiting from the
  volume of test queries this session, never to a permission or
  correctness defect. Zero unauthorized citations or incorrect refusals
  occurred at any point.
- The graph-extraction model (`deepseek/deepseek-v3.2`) is measurably flaky
  for structured extraction (6/12 in-body-502 failures in a live probe), but
  this is now contained rather than fatal: malformed-output and ontology
  errors are retried automatically, and OpenRouter model fallbacks reroute to
  a healthy model when the primary's providers fail. A production model swap
  is optional now, not required.
